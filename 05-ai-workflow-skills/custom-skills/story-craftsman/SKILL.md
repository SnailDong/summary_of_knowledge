# Story Craftsman

## 适用场景

使用本 skill 当用户需要：

- 创建、改写或审查 User Story、PRD 补充、AC、验收标准。
- 将一句话需求转成可设计、可测试、可 review 的范围。
- 修复已有需求文档：AC 不可测、范围不清、混入技术细节、缺非范围。
- 为后续架构、测试、可观测性建立追溯入口。

## 不适用场景

不使用本 skill 当：

- 纯内部重构且没有用户可见行为变化。
- 小型 bugfix 已有明确复现、期望和测试入口。
- 用户明确只要脑暴，不要文档产物。

## 触发场景

当用户说出类似下面的话时触发：

- “帮我写一个用户故事：用户可以给设备延保。”
- “把这个需求整理成 US 和 AC。”
- “这个 PRD 太乱，帮我补范围、非范围和验收标准。”
- “这个功能要进入架构设计前，先把需求边界梳理清楚。”
- “检查一下这个用户故事是不是可测试。”

## 依赖 Skill

无强依赖。该 skill 的输出通常作为 `architect` 和 `dev-workflow` 的输入。

## 使用契约

这个 skill 负责把模糊需求转成稳定、可引用、可测试的需求资产。它不写技术方案，不把实现细节塞进用户故事，不替用户猜业务事实。

## 编排关系

本 skill 不主动编排其他 skill。它的产物用于后续阶段：

- 交给 `architect`：作为技术设计、ADR、API/schema、状态机和测试观测设计的输入。
- 交给 `dev-workflow`：作为完整研发流程中 Intake 阶段的产物。
- 交给 `code-review`：作为合并前审查需求一致性的证据。

## Workflow：需求整理 1-7

本 skill 按 1-7 的编号顺序处理需求。除非用户明确只要求审查已有文档，否则不要跳过事实采集和范围确认。

| 环节 | 目标 | 联动 skill | 通过条件 |
|---|---|---|---|
| 1. 事实采集 | 确认角色、场景、痛点、目标 | 无外部调用 | 核心事实不靠猜测 |
| 2. 范围定界 | 区分本期、非范围、延后项 | 无外部调用 | 交付边界可复述 |
| 3. 业务分组 | 按用户能力组织故事 | 无外部调用 | 每组能对应一个用户价值 |
| 4. AC 编写 | 写出可测试验收条件 | 无外部调用 | 主路径、失败、边界可验证 |
| 5. NFR 和观测 | 补齐性能、安全、可观测性 | 后续交给 `architect` 细化 | 用户结果和失败可观察 |
| 6. 状态与索引 | 标注文档状态并接入入口 | 后续交给 `dev-workflow` 编排 | 后续设计能引用 |
| 7. 测试映射 | 把 AC 连接到验证思路 | 后续交给 `code-review` 复核 | 已实现 AC 有测试方向 |

## 1. 需求事实采集

生成或修改文档前，先建立“事实表”。不要边写边猜。

| 事实 | 必须回答 | 可接受输入 |
|---|---|---|
| 角色 | 谁使用、谁受影响、谁运营 | 用户口述、PRD、issue |
| 场景 | 用户何时触发需求 | 用户旅程、页面、任务流 |
| 痛点 | 当前问题造成什么损失 | 数据、反馈、客服、体验描述 |
| 目标 | 做完后什么变好 | 指标、行为变化、可观察结果 |
| 本期范围 | 这次交付什么 | 功能清单、流程、角色 |
| 非范围 | 明确不做什么 | 延后项、依赖条件 |
| AC | 怎么验收 | Given/When/Then、检查项、测试想法 |
| 风险 | 哪些场景容易失败 | 权限、空态、弱网、重复、并发、兼容 |

缺任何核心事实时，一次性提问。提问模板：

```markdown
为了把需求整理成可开发、可测试的用户故事，我还缺这些信息：

1. 用户/角色：
2. 场景/触发点：
3. 目标/成功指标：
4. 本期范围：
5. 明确不做：
6. 边界和失败场景：
7. 你认为“完成”的验收现象：
```

## 2. 产物位置契约

默认路径：

```text
docs/product/user-stories/{feature-slug}.html
```

若项目已经统一使用 Markdown，可使用 `.md`，但仍要有索引、状态和测试映射。

命名规则：

- 小写 kebab-case。
- 名称表达业务能力，例如 `device-sharing.html`。
- 不使用 `new-feature`、`story`、`requirement`、日期或个人名。

模块归属判断顺序：

1. 用户明确指定的模块或业务域。
2. 当前仓库和目录。
3. 已有 diff 的顶层目录。
4. 现有 user story 文件命名。
5. 仍不确定时询问。

跨端/跨服务需求可放在一个业务能力文件里，但必须在概览和状态表写清涉及端、服务和模块。

## 文档骨架

新建或大幅重写时使用以下结构。旧文档小改不强制迁移。

```markdown
# 用户故事：<功能名>

## 1. 背景
说明用户、场景、痛点、为什么现在要做。

## 2. 范围摘要
| 项目 | 内容 |
|---|---|
| 本期包含 |  |
| 本期不做 |  |
| 主要角色 |  |
| 成功指标 |  |
| 风险提示 |  |

## 3. 业务视角概览
| 分组 | 用户/角色视角 | 包含 US | 优先级 | 状态 |
|---|---|---|---|---|

## 4. <分组：能力名（角色视角）>
### US-XX-01：<标题>
作为 <角色>，我希望 <行为>，以便 <目标>。

#### 验收标准
- Given <前置条件>，When <触发动作>，Then <可观察结果>。

## 5. Future / Later
- 已识别但本期不做的方向、触发条件。

## 6. 非功能需求
- 性能：
- 可用性：
- 兼容性：
- 安全/隐私：

## 7. 可观测性需求
- 入口曝光：
- 关键动作：
- 成功事件：
- 失败事件：

## 8. 实现状态
| US | 状态 | 涉及端/模块 | 说明 |

## 9. 验收与测试映射
| AC | 验证方式 | 测试文件/用例 | 状态 |

## 10. 待确认
- [ ] ...
```

## 3. 业务分组组织法

当功能满足任一条件，必须按分组组织：

- 多个用户角色。
- 用户、运营、审核、客服、管理员等视角不同。
- 同一能力横跨 App/Web/Backend/Admin。
- 本期和未来能力容易混在一起。
- 有多个关键路径或状态阶段。

分组应表达稳定业务结构，不用优先级当章节。优先级只作为表格字段。

## 4. AC 编写质量标准

合格 AC 必须满足：

- 前置条件明确。
- 触发动作明确。
- 预期结果可观察。
- 能转成测试或人工验收。
- 覆盖主路径和关键边界。

至少考虑：

- 空数据。
- 权限不足。
- 重复操作。
- 网络或外部依赖失败。
- 旧数据或老客户端。
- 并发或重复提交。
- 取消、回退、重试。

不合格示例：

- “系统支持。”
- “页面显示正常。”
- “性能较好。”
- “逻辑正确。”

合格示例：

- “Given 用户没有任何设备，When 打开设备列表，Then 页面展示空态说明和添加入口。”
- “Given 提交请求超时，When 用户点击重试，Then 保留原输入并重新发起请求。”

## 6. 状态与索引

维护索引：

```text
docs/product/user-stories/index.html
```

索引字段：

| 字段 | 说明 |
|---|---|
| 功能 | 文档标题 |
| 文件 | 相对链接 |
| 主要角色 | 用户或 stakeholder |
| 状态 | 待开始/进行中/已实现/已验证/暂缓 |
| 更新时间 | 最近一次变更 |

状态只能使用：

- 待开始
- 进行中
- 已实现
- 已验证
- 暂缓

不要使用“差不多”“基本完成”“后续看”。

## 7. 测试映射

每条 AC 至少有一条映射：

| AC | 验证方式 | 状态 |
|---|---|---|
| AC-1 | L3 E2E / API E2E / 集成测试 / 人工验收 | 待补充 / 已覆盖 / 不适用 |

规则：

- 用户可见行为优先映射到 L3/E2E 或人工验收。
- 后端/API 能力映射到接口、集成或契约测试。
- 暂不能自动化时，写人工步骤和原因。
- 已实现 AC 不允许没有测试或验收映射。

## 附录 A：初始化规则

若项目没有需求目录，创建：

```text
docs/product/user-stories/
docs/architecture/
docs/testing/
docs/deployment/
docs/user-guide/
```

若没有文档目录说明，创建或更新 `docs/AGENTS.md`，只写目录用途和维护规则，不放长篇需求正文。

## 附录 B：旧文档修订策略

| 变更 | 动作 |
|---|---|
| 文案修正 | 原地小改 |
| 新 AC | 更新 AC 和测试映射 |
| 新角色/路径 | 新增分组，更新概览 |
| 范围调整 | 同步本期范围、非范围、Future |
| 文件重命名 | 更新索引和相对链接 |

## 附录 C：退回条件

出现任一项，不能交给架构设计：

- 关键业务事实靠猜。
- 混入 API、DB、类名、组件名、任务拆分。
- AC 无法验证。
- 没有非范围。
- 只写 happy path。
- 多角色需求未分组。
- 新建文档没有稳定路径或索引。
- 已实现 AC 没有测试/验收映射。

## 附录 D：交付清单

- 事实表完整。
- 范围和非范围明确。
- AC 可测试。
- 没有技术实现细节。
- 分组结构合理。
- 状态和测试映射齐全。
- 索引已同步。
- 待确认问题列出。

## 附录 E：访谈问题库

当信息不足时，从下面问题中选择相关项，一次性询问。不要逐条追问。

角色与场景：

- 谁是发起动作的人？
- 谁会看到结果？
- 谁负责处理失败或异常？
- 这个需求发生在用户旅程的哪一步？
- 用户进入这个能力前已经完成了什么？

目标与指标：

- 做完后用户少做了什么、快了什么、避免了什么？
- 业务侧希望哪个指标变化？
- 如果只能选一个成功指标，会选哪个？
- 哪些指标只能观察，不能作为成功判断？
- 什么情况说明这个需求失败了？

范围与非范围：

- 本期必须包含哪几个动作？
- 哪些能力已被讨论但明确不做？
- 哪些能力依赖外部系统、设计稿、运营配置或数据准备？
- 本期是否需要后台、运营、客服或管理员能力？

边界：

- 空数据时用户看到什么？
- 权限不足时怎么提示？
- 用户重复点击或重复提交时怎么处理？
- 网络失败或服务失败时是否保留输入？
- 老版本客户端或历史数据是否需要兼容？
- 取消、返回、刷新、重试分别发生什么？

验收：

- 哪个页面、接口、日志、数据或测试能证明完成？
- 哪些场景必须自动化？
- 哪些场景可以人工验收？
- 验收失败时如何定位是产品问题、实现问题还是数据问题？

## 附录 F：分组判定表

用这张表判断是否需要业务视角分组。

| 信号 | 分组方式 |
|---|---|
| 作者/阅读者/运营看到不同体验 | 按角色分组 |
| 创建/审核/发布/消费多个阶段 | 按流程阶段分组 |
| App/Web/Backend/Admin 同时参与 | 按用户路径分组，表中标注模块 |
| 本期和未来能力容易混淆 | 本期分组 + Future 分组 |
| 失败处理和成功路径同等重要 | 成功路径分组 + 异常处理分组 |

分组数量不是越多越好。一个分组必须有独立用户价值或独立验收意义。

## 附录 G：AC 编写细则

每条 AC 优先写成：

```text
Given <前置条件>
When <用户动作或系统触发>
Then <用户、数据、日志或系统可观察结果>
```

可接受的 Then：

- 页面出现某个状态。
- 数据产生或不产生某个变化。
- 用户收到明确反馈。
- 操作被阻止并说明原因。
- 事件、日志或指标出现。
- 下游任务被排队、取消、重试或补偿。

不可接受的 Then：

- “成功处理。”
- “逻辑正确。”
- “调用接口。”
- “写入数据库。”
- “代码覆盖。”

技术词只允许出现在测试映射或备注中，不允许放入 US 主体。

## 5.1 NFR 编写细则

NFR 必须可判断，不写空泛形容词。

| 类型 | 写法 |
|---|---|
| 性能 | p95/p99、最大等待时间、列表规模 |
| 可用性 | 失败后的降级体验、重试、恢复 |
| 兼容性 | 老客户端、旧数据、旧配置 |
| 安全/隐私 | 权限、最小展示、敏感字段不可见 |
| 可访问性 | 键盘、读屏、对比度、触控区域 |
| 国际化 | 文案长度、占位符、RTL 或多语言限制 |

如果暂时没有量化值，写“待确认”并列入 Open Questions，不要写“性能较好”。

## 5.2 可观测性需求写法

需求阶段不设计完整观测方案，但要写出产品侧希望知道什么。

至少回答：

- 用户是否看到入口？
- 用户是否开始关键动作？
- 用户是否完成？
- 用户在哪一步失败或放弃？
- 失败原因是否需要区分？
- 是否需要知道不同平台、版本、实验分组的差异？

示例：

| 问题 | 需求表达 |
|---|---|
| 入口曝光 | 需要知道多少用户看到入口 |
| 点击转化 | 需要知道看到入口后有多少用户点击 |
| 完成 | 需要知道点击后有多少用户完成 |
| 失败 | 需要区分用户取消、权限失败、网络失败、服务失败 |

## 附录 I：Review 自查表

交付前用下面表格自审：

| 检查项 | 通过条件 |
|---|---|
| 用户视角 | 每条 US 都能读出“谁要什么、为什么” |
| 可测试 | 每条 AC 都能转成测试或验收 |
| 非范围 | reader 能知道本期不做什么 |
| 分组 | 多角色/多路径需求没有混成一团 |
| 状态 | 每条 US 状态明确 |
| 追溯 | AC 能追溯到测试或验收方式 |
| 观测 | 用户可见行为有基本观测需求 |
| 索引 | 新文档可从 index 找到 |

## 附录 J：常见修复模式

| 问题 | 修复 |
|---|---|
| AC 写成“系统支持 X” | 改成 Given/When/Then |
| US 写了 API 或表字段 | 移到技术设计或测试映射 |
| 只有 happy path | 补失败、空态、权限、重复操作 |
| P0/P1 做章节 | 改为分组表中的属性 |
| 多端需求散在多个文件 | 建业务总文件，模块细节放状态表 |
| 已实现无测试映射 | 补验收与测试映射表 |
