## 一、Flutter 设备绑定模块重构

### 背景（30 秒开场）

> "我们的智能家居平台 Vico 做到 4.0 版本时，设备绑定模块已经积累了大量历史包袱——权限管理、导航路由、蓝牙状态逻辑全部耦合在一起，每次改需求都要小心翼翼，Bug 频发。VH 4.0 要支持 Matter 新协议，老架构完全撑不住，所以我主导了这次全面重构。"

---

### 难点一：蓝牙权限与蓝牙状态的错误区分

**面试官常问**：*遇到过什么印象深刻的 Bug？*

**回答思路**：

问题根源：Android 上「蓝牙未开启」和「蓝牙权限未授权」在底层返回的错误码是一样的，原来的代码没有区分这两种情况，统一弹一个提示，但引导方向完全不同——

- 蓝牙未开启 → 应该引导用户去打开蓝牙开关
- 权限未授权 → 应该引导用户去系统设置开权限

结果就是用户明明权限有了，但蓝牙没开，点提示进去看到的是权限页面，完全懵了。

**解决方案**：

设计了一个 Permission Guard，双轨并行监听：
1. 一条轨道监听运行时权限状态（`checkSelfPermission`）
2. 另一条轨道监听蓝牙适配器状态（`BluetoothAdapter.STATE_ON/OFF`）

两个状态独立判断、独立 UI 提示，互不干扰。另外加了一个权限撤销后自动重检机制——用户从设置页回来之后，不用手动重试，自动触发一次权限检查，体验顺滑很多。

**效果**：绑定失败中，由于提示错误导致用户放弃的情况明显减少。

---

### 难点二：多路绑定路径的导航混乱

**面试官常问**：*怎么做路由管理的？遇到过返回栈异常吗？*

**回答思路**：

老代码里套装设备、主设备、Matter 设备共用一套导航栈，不同路径在历史迭代中打了很多补丁，出现了嵌套跳转——A 页面跳 B，B 又跳回 A 的某个子页面，返回键行为完全不可预期。

**解决方案**：

引入 SingleTask 模式，核心思路是：每种设备类型有且只有一个绑定任务栈，进入绑定流程时清空当前栈，保证导航状态干净。三条路径（套装 / 主设备 / Matter）完全隔离，互相不影响。

Matter 比较特殊，因为需要动态下架上架（有些市场不支持），所以用枚举存根管理，主流程代码里不硬编码 Matter 的逻辑，通过注册机制插拔，不影响主流程稳定性。

---

### 难点三：Flutter ↔ Native 通道碎片化

**面试官常问**：*Flutter 和原生是怎么通信的？碰到过什么坑？*

**回答思路**：

原来有一个自定义的 NativeBridge 层，各种方法七零八落地定义在里面，Android 端和 iOS 端实现不完全一致——比如同一个回调，Android 返回的是 JSON 字符串，iOS 返回的是 Map，Flutter 层要做两套解析，每次改动都要同时改三个地方，很容易漏。

**解决方案**：

全部迁移到标准 MethodChannel（`plugin.channel`），制定了明确的 API 契约文档，参数格式、错误码、回调结构全部统一。Android/iOS 各自实现同一套接口，Flutter 层只写一份解析逻辑。

改完之后，新增一个绑定能力，Android/iOS 的改动量从原来各自独立实现减少到只需要实现同一套接口定义，效率提升明显。

## 二、数据脱敏框架

### 背景

> "设备绑定过程中会采集 WiFi 密码、MAC 地址、IMEI 这些敏感信息用于问题诊断和埋点。欧盟 GDPR 和国内个保法都有明确要求，这些数据不能明文上报。"

---

### 原始方案的问题

**面试官常问**：*为什么要自己造这个轮子？*

**回答思路**：

最开始的做法是在每个上报点手写脱敏：`password.replace(...)` 这种。问题很明显——

1. 规则散落在几十个文件里，新来的人根本不知道哪里需要脱敏
2. 加新字段时容易忘，某次上线前安全 Review 就发现漏掉了两个字段
3. 不同业务场景需要不同脱敏策略（测试环境要能看明文，生产要完全屏蔽），手写根本没法灵活配置

---

### 解决方案

**核心设计**：配置驱动 + 零业务侵入

设计了一个 JSON 配置文件，所有敏感字段的识别规则和处理策略都在这里定义：

```json
{
  "keyBased": [
    { "key": "password", "strategy": "CLEAN" },
    { "key": "ssid", "strategy": "MASK", "keepPrefix": 2 }
  ],
  "valueBased": [
    { "pattern": "MAC地址正则", "strategy": "MACADDRESS" },
    { "pattern": "IPv4正则", "strategy": "IP" }
  ]
}
```

双层规则：
- **KeyBased**：按字段名精准匹配，优先级最高
- **ValueBased**：按值的正则模式识别，兜底，防止字段名变了但值还是敏感数据

处理策略四种：CLEAN（直接删除）、MASK（部分掩码，比如保留前两位）、HASH（SHA-256，诊断时还能对比）、专用策略（MAC 地址、IP 各有规范格式）。

执行层用 Kotlin 扩展函数包装，业务代码只需要：

```kotlin
val safeData = bindModel.toJsonString().toSensitiveJson()
```

业务完全不感知脱敏过程。配置用单例 + AtomicBoolean 保证线程安全初始化，不会重复解析。

**效果**：
- 新增敏感字段只改 JSON，不动业务代码
- 一次安全 Review 再没发现遗漏
- 测试/生产环境加载不同配置，调试效率没有损失

## 三、FRP 专线加速与智能降级

### 背景

> "直播场景下，用户在跨运营商网络或偏远地区看摄像头，延迟能到 3-5 秒，基本没法用。我们引入了 FRP 专线加速，但 FRP 节点本身不稳定，处理不好反而会让体验更差。"

---

### 核心难点：FRP 失败后的级联故障

**面试官常问**：*怎么做降级的？*

**回答思路**：

最初的实现是 FRP 失败就重试，重试失败再重试，结果用户看到的是长时间黑屏，体验比没有 FRP 还差。而且一个设备 FRP 不通，会影响到用户设备列表里其他设备的连接判断，级联故障。

**解决方案**：

设备级失败计数机制——每台设备独立维护一个 FRP 失败计数器：

```
设备A FRP失败次数: 0 → 1 → 2（禁用）
设备B FRP失败次数: 0（不受影响）
```

连续失败 2 次，该设备自动禁用 FRP，直接走 SafeRTC 降级通道，用户无感知。下次用户主动刷新设备列表时重置计数，给 FRP 一次重新尝试的机会。

另外不同场景做了差异化处理：
- 直播播放：优先走 FRP，失败降级
- 心跳包：不走 FRP（心跳没有 body，走专线没有意义）
- 录像下载：优先 FRP 地址
- API 调用：部分豁免，根据接口类型配置

**效果**：FRP 可用时延迟从 3-5 秒降至 200-400ms；FRP 不可用时自动降级，用户不会长时间卡屏。iOS SDK / Android SDK / Flutter 三端统一实现，策略一致。

## 四、iOS 绑定模块拆分（SmartDeviceCoreSDK）

### 背景

> "iOS 绑定模块的 BindDeviceViewModel 文件有 526 行，QR 码绑定和 4G 绑定的逻辑全耦合在里面。iOS 14 之后苹果新增了 Local Network 权限要求，在这个单体文件里加权限逻辑，改一处很容易影响另一处。"

---

### iOS 14 Local Network 权限问题

**面试官常问**：*遇到过 iOS 系统版本兼容问题吗？*

**回答思路**：

iOS 14 开始，Local Network 访问（mDNS 服务发现）需要弹权限弹窗。我们的 4G 绑定流程依赖 mDNS 来发现设备，QR 码绑定走 BLE 直连，不需要 mDNS。

问题在于，如果在进入绑定页面时统一请求 Local Network 权限，QR 码用户会看到一个莫名其妙的权限弹窗——"允许访问本地网络？"——他们不知道这是为什么，大概率点拒绝，然后 4G 绑定就用不了。

**解决方案**：

把 QR 码绑定流程和 4G 绑定流程彻底分离，各自一个 Flow 类。QR 码流程完全不涉及 mDNS，不触发 Local Network 权限弹窗。4G 流程进入时才调用 `checkPermissionOnce`——

这个方法的设计要点是"单次"：如果用户已经授权过，不再弹窗；如果用户取消了弹窗（比如手势关闭），2 秒后自动恢复流程，不卡死。iOS 14 以下直接跳过这步。

**模块拆分结果**：
- 526 行单体 → 按 Discovery / Transport / Business 三层，每个文件控制在 100 行以内
- `BindResultPoller` 单独抽出来处理绑定结果轮询，支持自定义超时和重试策略
- QR 码 Flow 和 4G Flow 各自独立，后续单独修改不互相影响

## 五、HB1 解绑保留配对关系方案

### 背景

> "用户买了一套摄像头套装，Hub 加 4 个 Camera，出厂时已经预配对好了。但用户如果解绑 Hub 再重新绑，预配对关系只有首次有效，4 个 Camera 要逐个手动重新配对，用户体验很差。产品希望解绑时给用户一个选项：保留还是清除配对关系。"

---

### 核心设计挑战

**面试官常问**：*如何设计一个跨多端的功能？*

**回答思路**：

这个需求涉及 App、Cloud、Device 固件三端，任何一端的决策都会影响其他两端，所以先做了一次技术方案拉通。

几个关键决策点：

**决策 1：配对关系存在哪里？**

- 方案 A：存 Cloud。优点是 Hub 重置也能恢复，缺点是离线时不可用，而且 Cloud 要长期维护这份数据
- 方案 B：存 Device 本地（NVS/Flash）。离线可用，设备自治，Cloud 只在重新绑定时同步一次

选了方案 B，核心理由是：配对关系是设备本地状态，应该由设备自己管理，不应该依赖 Cloud 的持久化。

**决策 2：App 怎么拿到配对列表？**

最直接的想法是通过 BLE 直接从 Hub 读取。但 BLE 带宽有限，传输完整列表效率低，而且需要保持 BLE 连接状态。

更优的方案：Hub 绑定成功连上 Cloud 之后，主动把本地配对列表推送给 Cloud，App 再从 Cloud 用**现有的子设备查询接口**来读。这样 App 端几乎不用改，完全复用预绑定的读取流程，改动量最小化。

**决策 3：BLE 广播怎么告知 App？**

Hub 在待绑定状态广播时，新增一个 `has_pairing` bit 位，App 扫描到设备后就知道要走哪条绑定路径：
- `resetkey=0`：走预绑定流程（出厂首次）
- `has_pairing=1`：走配对保留恢复流程
- 都不是：走手动配对流程

**兼容性设计**：

这是最容易出问题的地方。旧 App 不传 `clear_pairing` 参数 → Cloud 默认 `true`（清除），行为和原来一致；旧固件不认识 `clear_pairing` → 默认清除，也和原来一致。新旧交叉的 4 种组合都定义了明确的行为，不会出现功能异常。

## 六、SCLogAnalysisTool 日志分析工具

### 背景

> "做 IoT 嵌入式开发，经常要分析几百 MB 的设备日志。公司 Windows 同事用 Notepad++，我们 Mac 用户没有好用的同类工具，每次分析日志都很低效。我就用 Python 自己写了一个。"

---

### 技术亮点

**面试官常问**：*个人项目里你觉得技术上有意思的地方是什么？*

**回答思路**：

**① 大文件加载性能**

几百 MB 的日志文件如果一次性读进内存，界面会卡死。用了分段加载策略——先加载前 10000 行展示，用户滚动到底部再按需加载下一段，同时用 `QSyntaxHighlighter` 异步高亮，不阻塞 UI 线程。

**② 三区联动跳转**

这是用户体验的核心：过滤面板双击一条日志，完整日志面板自动跳转到对应的原始行，并且垂直+水平居中显示。实现上需要过滤结果记录每条日志在原始文件中的行号，用 Qt 的信号槽机制在两个面板之间同步状态。

**③ 关键词持久化**

工作中反复用到的关键词（比如设备 SN、特定错误码）保存为分组，下次打开工具直接点击就能应用到过滤条件，不用每次重新输入。配置用 JSON 持久化，工具关闭后不丢失。

**架构**：MVC 分层，FileManager / FilterEngine / KeywordManager 各自独立，改一个模块不影响其他，最终打包为 macOS App（.app bundle）方便分发。

## 七、IoT 嵌入式系统开发模板

### 背景

> "团队做嵌入式开发的一个痛点是：必须有真实硬件才能调试，硬件不够用，而且硬件调试循环很慢。我设计了一套开发模板，让 80% 的问题能在浏览器里就发现和解决。"

---

### 核心方案

**面试官常问**：*怎么提升嵌入式开发效率的？*

**回答思路**：

核心思路是把 C++ 设备代码编译成 WebAssembly，在浏览器里跑一个仿真环境。React 写设备控制 UI，和 Wasm 模块交互，模拟真实设备的行为。

这样就有了三层测试体系：
- **L1**：C++ 层的 GTest 单元测试，测业务逻辑（不依赖硬件）
- **L2**：HAL Mock 接口测试，把硬件 GPIO 换成 Mock 实现，测设备层
- **L3**：Playwright 浏览器 E2E 测试，测完整的用户交互流程

大部分 Bug 在 L1/L2 就能发现，L3 发现集成问题，到真实硬件上基本就是验证，不是调试。

架构分层是 Cluster（业务逻辑，可复用）→ Device（组装 Cluster，实现 HAL）→ Service（入口，注册命令处理器），职责清晰，换一个设备只需要换 HAL 实现。

## 八、常见面试问题参考答案

### Q：你最大的技术挑战是什么？

> 推荐用**数据脱敏框架**或**FRP 降级机制**，因为这两个有明确的问题 → 设计决策 → 量化效果的完整链路，讲起来结构清晰。

---

### Q：讲一个你主导的架构设计

> 推荐用 **HB1 解绑保留配对关系方案**，这个涉及三端协作，有多个关键设计决策（存哪里、怎么传、怎么兼容），能体现系统设计能力。

---

### Q：如何保证代码质量？

- 代码层面：SonarQube 扫描、pre-commit hook、严格的 Conventional Commits 规范
- 测试层面：SDK 层 JUnit 单元测试，嵌入式项目 L1/L2/L3 分层测试
- 流程层面：GitLab MR + Code Review，测试通过才能合并

---

### Q：怎么处理 Android/iOS 双端差异？

> 以**权限守卫**为例：设计统一的抽象层（Permission Guard 接口），Android/iOS 各自实现，Flutter 层只调用抽象接口。需要平台特定处理的（比如 iOS 14 Local Network 权限）在各端封装好，不暴露给上层。

---

### Q：有没有做过性能优化？

几个可以展开讲的点：
1. 脱敏框架：单例缓存配置，避免每次上报都重新解析 JSON
2. 日志工具：大文件分段加载 + 异步高亮，避免 UI 卡死
3. FRP 失败计数：防止无效重试消耗资源
4. 蓝牙通道统一：减少双端维护成本，间接提升开发效率
