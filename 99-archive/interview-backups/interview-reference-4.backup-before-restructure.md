**回答：**

我会要求双端 commit 语义一致、同日合入或按明确顺序合入，避免出现 Flutter 已经调用新参数但某端 Native 还没实现的情况。

### Q14：跨仓协作最容易出什么问题？

**回答：**
最容易出三类问题：  
第一，契约漂移，比如 Android 返回一种结构、iOS 返回另一种结构。  
第二，版本错位，比如 Flutter 上层已经发版，但 Native SDK 还没跟上。  
第三，测试覆盖不一致，一端只测 happy path，另一端补了边界。  
我的做法是把契约、错误码、参数、固件门控和验收路径写清楚，用同一套测试清单和 review 规则推进。

### Q15：AI 辅助研发的产物怎么管理？

**回答：**
不是让 AI 直接随便提交。我的做法是先写架构约束和任务边界，比如 CLAUDE.md、ADR、quality gate、测试要求；然后 AI 按小任务执行；提交后通过 review、测试、文档漂移检查和 golden data 进行验证。  
我更关注的是 AI 产物是否可审计、可归因，也就是每个变更为什么做、对应哪个需求或 ADR、有没有测试和验证证据。

### Q16：AI 写出来的代码质量怎么保证？

**回答：**
我把 AI 常见失败模式拆成几类：只做 happy path、漏测试、文档代码漂移、过度设计、误用已有接口、隐藏失败。  
对应的治理方式是：  
一是架构约束前置，比如 Ports & Adapters、Split API 的边界写进规则；  
二是测试和 golden data，关键输入输出必须可对照；  
三是多轮 review，先看需求/设计，再看代码行为，再看测试和观测；  
四是 quality gate，把必须满足的检查变成脚本或 CI 门禁。

## 4. Provisioning Kit 项目相关

### Q20：Provisioning Kit 的项目背景是什么？

**回答：**
设备绑定能力原来散落在宿主 App 和不同仓里，UI、网络、权限、导航、埋点、Native 配网能力耦合在一起。新消费端想复用时，要复制大量代码，还容易引入不一致。  
Provisioning Kit 的目标是把设备配网/绑定能力抽成独立 SDK，让不同 App 只实现少量宿主 Adapter，就能复用同一套绑定能力。

### Q21：你说“六边形架构 + Split API”，具体怎么设计？

**回答：**
Split API 是把接口和值对象放在 API 包，把具体业务实现放在实现包。API 包尽量零运行时依赖，避免把 UI、网络、埋点、Native 依赖传染给消费方。  
六边形架构是把外部基础设施通过 Port 注入，比如 HTTP、权限、导航、i18n、埋点。SDK 内部只依赖抽象 Port，不直接依赖宿主框架。  
这样新 App 接入时，不需要复制 SDK 内部逻辑，只需要实现宿主 Adapter。

### Q22：为什么要强调 API 包零运行时依赖？

**回答：**
因为 API 包是所有消费方都会依赖的最小公共面。如果 API 包引入 UI、网络库、状态管理或宿主框架，消费方即使只想调用一个只读能力，也会被迫引入一堆依赖。  
零运行时依赖能让 API 包稳定、轻量，也降低版本冲突和接入成本。

### Q23：5 个宿主 Port 分别解决什么问题？

**回答：**
它们负责把 SDK 和宿主基础设施隔离开。  
HTTP Port 解决网络请求不绑定宿主网络库。  
权限 Port 解决蓝牙、定位、系统权限在不同端上的差异。  
导航 Port 解决 SDK 页面跳转不直接依赖宿主路由。  
i18n Port 或资源注入解决文案多语言。  
埋点 Port 解决 SDK 不直接绑定 Snowplow 或宿主埋点系统。  
核心思想是 SDK 定义需要什么，宿主负责怎么实现。

### Q24：native 配网核心三代演进为什么不是反复推翻？

**回答：**
关键看哪一层稳定、哪一层变化。三代变化主要发生在底层 native adapter：从 channel，到 Dart FFI，再到 native AAR + session API。  
但是上层 Port 契约和 SDK 业务模型尽量保持稳定。也就是说，外部消费方看到的是稳定的 SDK API，底层实现可以替换。  
这正是六边形架构的价值：可替换的是 adapter，不是整体业务模型。

### Q25：Dart FFI 和 native AAR 各有什么取舍？

**回答：**
Dart FFI 的优势是更直接、更低层，但它对跨端构建、符号、平台差异、调试链路要求更高。  
native AAR + session API 更符合 Android 原生能力复用和版本发布，能把底层配网能力以更稳定的 native artifact 交付。  
最终选择不是说 FFI 不好，而是结合团队维护成本、跨端稳定性和发布链路，AAR + session API 更适合生产演进。

### Q26：什么是 session API？

**回答：**
session API 是把一次配网过程抽象成有生命周期的会话，而不是零散调用。比如 createSession、prepare、fetchWifiList、commitBind 这类步骤都围绕同一个 session。  
好处是状态更清楚，错误和超时能归属到具体会话，也方便后续做取消、重试、日志和诊断。

### Q27：Provisioning Kit 最大难点是什么？

**回答：**
最大难点不是单个页面，而是解耦。设备绑定原来依赖宿主网络、权限、导航、埋点、Native 通道和业务状态，如果直接搬进 SDK，只是换个目录，不能真正复用。  
所以难点是识别哪些是 SDK 内部稳定能力，哪些是宿主特定能力，然后用 Port/Adapter 隔离。另一个难点是分阶段迁移，保证每一步都有可验证退出条件，不影响现有 App 绑定链路。

### Q28：如何证明这个 SDK 化有价值？

**回答：**
从接入成本看，30+ 配网页面和绑定逻辑被内聚进 SDK，新消费端不再复制页面和流程，只需要实现少量 Adapter。  
从演进能力看，底层 native 方案换了三次，上层契约能保持稳定，说明边界设计有效。  
从交付沉淀看，53 天完成三代演进和 22 个 ADR，后续团队能理解为什么这样设计。

### Q29：如果新 App 要接 Provisioning Kit，你会怎么指导？

**回答：**
我会先确认宿主能力：网络请求、权限申请、导航、埋点、主题/i18n、Native artifact 是否可用。  
然后按 API 包接入，先跑 example 或 fake 装配，确认基础流程。  
最后实现宿主 Adapter，并做绑定主路径、权限拒绝、蓝牙未开、WiFi 列表失败、绑定超时等场景测试。  
我的原则是先跑通最小闭环，再补边界和观测。

## 5. Homebase 新型号项目相关

### Q30：Homebase 新型号功能的业务背景是什么？

**回答：**
Homebase 是新的中枢设备，绑定模型从原来“相机直接绑定到 App/云”变成“相机绑定到 Hub”的 1 对 N 拓扑。  
这带来了新页面、新绑定链路、固件版本门控、在线/离线换网、解绑是否保留配对等一系列新能力。  
我的工作是把这套能力在 Flutter、Android、iOS、iOS Core SDK 四个仓里协同落地。

### Q31：为什么这个项目要 4 仓锁步？

**回答：**
因为 Flutter 上层页面要调用 Android/iOS Native 能力，iOS 又涉及 Core SDK，任意一个仓没跟上都会导致某端不可用。  
比如 Flutter 新增一个 MethodChannel 参数，如果 Android 实现了、iOS 没实现，就会出现双端行为不一致。  
所以必须先定义统一契约，再四仓按同一节奏实现和合入。

### Q32：Permission Guard 解决什么问题？

**回答：**
它主要解决蓝牙状态和权限状态混淆的问题。  
在 Android 上，蓝牙未开启和蓝牙权限未授权对用户来说是两种完全不同的引导：一个要打开蓝牙开关，一个要去系统设置授权。  
Permission Guard 会把系统权限和蓝牙开关状态分开判断，避免给用户错误提示。

### Q33：SingleTask 隔离绑定导航是什么意思？

**回答：**
设备绑定流程很长，可能从扫码、选择 Hub、选择 WiFi、固件校验、BLE 连接、云端绑定一路跳转。如果多个绑定路径共用一个混乱返回栈，返回键和异常恢复会不可控。  
SingleTask 的思路是每次进入绑定任务时建立清晰的任务栈，避免旧页面残留影响新流程。  
这对 IoT 绑定很重要，因为用户中途失败、返回、重试很常见。

### Q34：固件版本门控为什么重要？

**回答：**
新硬件能力通常依赖特定固件版本。如果旧固件不支持新协议或新参数，强行下发可能会导致绑定失败，甚至设备进入异常状态。  
所以我们在列表展示和绑定前都做固件版本校验：列表过滤是用户体验，绑定前校验是正确性保障，防止缓存或绕过导致错误。

### Q35：为什么在线和离线换网要走不同路径？

**回答：**
在线设备能通过云端或 WebSocket 信令触发换网，离线设备没有云端连接，只能通过 BLE 近场能力扫描 WiFi 并重新配置。  
两者的物理条件不同，强行统一会让流程更复杂，也可能引入不必要失败。  
所以在线走云端信令，离线走 BLE，这样更符合设备真实状态。

### Q36：解绑保留配对是什么场景？

**回答：**
用户解绑 Hub 或相机时，有些场景希望清空所有本地存储和配对信息，有些场景希望保留配对，方便后续重新添加或迁移。  
所以解绑接口需要支持不同参数组合，比如是否 retainPairing、是否 cleanStorage。  
难点是要保证旧客户端兼容，同时新客户端能表达更细的解绑意图。

### Q37：Homebase 项目最大的风险是什么？

**回答：**
最大的风险是跨端不一致和固件兼容。  
跨端不一致会导致同一个设备在 Android 和 iOS 上表现不同；固件兼容问题会导致用户明明流程正确但设备不支持。  
所以我会优先做契约统一、固件门控、双端测试和发布节奏控制。

### Q38：你说统一 Flutter 与 Native 通道契约，实际收益是什么？

**回答：**
以前新增或修改一个 Native 能力，Android/iOS/Flutter 各自有不同桥接和解析逻辑，容易出现参数名不一致、错误码不一致、某端漏实现。  
统一 MethodChannel 契约后，Flutter 只面向一套参数、错误结构和回调模型，Android/iOS 按同一接口实现。  
实际收益是联调更可控、问题定位更快、跨仓改动时不容易遗漏，也更方便做测试清单和 review。

## 6. Warranty Plus 项目相关

### Q39：Warranty Plus 的项目背景是什么？

**回答：**
业务上，运营需要把用户在 Amazon 等渠道的订单号、App 账号和设备关联起来，用于保修延保、审核和后续分析。之前缺少这样的系统，运营无法在 App 侧闭环处理。  
所以 Warranty Plus 做的是一套售后保修能力：用户在 App 里提交保修申请，后端记录和校验，运营在审核后台处理，三 App 共用同一套 Flutter SDK。

### Q40：你在 Warranty Plus 里具体负责什么？

**回答：**
我负责整体方案和交付闭环：Flutter Split-API SDK、Go 服务、Next.js 审核后台、MySQL 数据模型、订单号校验、配置热替换、弱网缓存、埋点和监控。  
后端和 Web 实现主要通过 AI Agent 协作完成，但接口设计、数据模型、联调、测试、质量门禁和验收是我负责的。

### Q41：为什么 Warranty Plus 要做 Split API SDK？

**回答：**
因为三 App 要共用能力，但不同 App 的宿主环境、UI、主题、埋点、配置可能不一样。  
Split API 可以让稳定的接口和值对象独立出来，消费方只依赖最小公共面；具体 UI、缓存、网络、埋点等实现放在 feature 包里。  
这样既能复用核心能力，又不会把某个 App 的框架强绑定给所有消费方。

### Q42：Go 服务是你手写的吗？

**回答：**
准确说是 AI 辅助实现。我主导的是方案、接口、数据模型、质量约束和验收。  
我能读懂和设计 Go 服务的分层，比如 handler、service/logic、repo/adapter、MySQL schema、配置、监控和部署，但我不会把自己描述成长期手写 Go 的后端专家。  
这个项目的价值在于我把 AI Agent 纳入可控工程流程，完成了一个跨 Flutter、Go、Next.js 的业务闭环。

### Q43：为什么保修申请和审核要设计双表模型？

**回答：**
保修申请和审核是两个不同生命周期。申请记录偏用户侧事实，比如用户、设备、订单号、渠道、提交时间；审核记录偏运营处理，比如审核状态、审核人、审核意见、处理时间。  
拆成双表能让用户提交和运营审核解耦，避免一个表塞太多状态和冗余字段，也更方便后续审计和状态追踪。

### Q44：如果面试官追问后端权限边界，你怎么讲？

**回答：**
我会先讲原则：后端不能信任客户端随便传 userId、deviceId，必须明确身份和设备上下文从哪里来。  
如果身份由 Gateway、登录态或宿主服务提供，Warranty 服务就消费可信上下文，并在自己的业务边界内做保修申请、审核状态和重复提交校验。  
重点不是把所有校验都塞进一个服务，而是把身份、设备归属、业务登记和运营审核的责任边界讲清楚，并用接口契约和测试覆盖。

### Q45：订单号实时校验怎么做？

**回答：**
订单号校验分两类：格式校验和业务校验。  
格式校验可以根据不同渠道规则做正则或结构判断，实时给用户反馈；业务校验要结合后端已有数据，避免重复提交、非法状态或已审核记录。  
实时校验的价值是减少用户提交后才失败，也降低运营审核压力。

### Q46：弱网缓存解决什么问题？

**回答：**
售后保修入口和提交链路不能因为短暂网络问题直接白屏。弱网缓存可以缓存配置、渠道信息或用户输入，让用户在网络波动时仍有可恢复体验。  
同时要注意缓存不能让用户提交过期或错误数据，所以缓存需要有版本、过期策略和失败回退。

### Q47：配置热替换有什么价值？

**回答：**
售后保修业务经常涉及渠道、文案、开关、实验和审核规则。如果每次调整都要发 App 版本，效率太低。  
配置热替换可以让后端或配置平台控制入口、渠道和策略。  
但热配置也有风险，所以需要默认值、缓存、熔断和监控，避免配置异常导致 App 页面不可用。

### Q48：Warranty Plus 的业务价值怎么讲？

**回答：**
运营第一次获得了 Amazon 订单号、App 账号和设备之间的关联能力，可以做延保审核和后续分析。  
技术上，三 App 复用一套 SDK，避免每个 App 单独开发一遍，节省约 2/3 重复工作。  
接口上，通过减少冗余返回和配置下沉，响应体积减少约 62%，也提升了移动端体验。

### Q49：埋点和 Grafana 看板在这个项目里做什么？

**回答：**
主要用于观察 SDK 健康度、功能使用状况和异常问题，比如入口曝光、提交量、审核量、核心接口错误、失败原因、配置降级、不同 App/版本的分布等。  
Grafana 用于把这些指标可视化，帮助研发和运营判断：用户有没有提交、失败在哪一步、影响范围多大、是否需要在客诉前处理。

### Q50：GrowthBook 在这个项目里做什么？

**回答：**
GrowthBook 可以用于 feature flag 或实验控制，比如入口是否开启、不同用户群是否展示保修入口、不同策略是否灰度。  
在 App 侧要保证 GrowthBook 异常时有默认策略，不能因为实验平台异常导致主流程不可用。

### Q51：ArgoCD 和 GitLab CI 在你的项目里怎么用？

**回答：**
GitLab CI 负责构建、测试、静态检查、质量门禁等自动化验证。  
ArgoCD 偏部署侧，用 GitOps 的方式把配置或服务部署到对应环境。  
我的关注点是让代码、配置、测试、部署之间有闭环，避免“本地能跑，但 CI 或部署环境出问题”。

### Q52：Warranty Plus 最大风险是什么？

**回答：**
第一是信任边界，如果身份和设备归属处理不清，可能出现越权或错误登记。  
第二是多 App 复用，如果 SDK 设计不好，会被某个宿主 App 的框架污染。  
第三是 AI 辅助全栈交付，如果没有质量门禁，后端/Web 容易只完成 happy path。  
所以我分别用信任边界设计、Split API、测试和 quality gate 来控制。

## 7. 工作技能相关

### Q53：Flutter/Dart 你最熟哪些部分？

**回答：**
我最熟的是 Flutter 工程化和 SDK 化，而不只是页面开发。包括 Package 化、Monorepo/Workspace、MethodChannel、跨端插件、状态和路由边界、宿主能力注入、测试和多 App 复用。  
在业务上主要用在设备绑定、保修延保、设备卡片、配置页面等复杂链路。

### Q54：MethodChannel 你踩过哪些坑？

**回答：**
常见坑有：Android/iOS 返回结构不一致、错误码不统一、参数 optional/required 没定义清楚、异步回调生命周期不一致、Flutter 页面销毁后 native 还回调。  
解决方式是先定义契约，包括方法名、参数、返回结构、错误码和线程/生命周期约束，然后双端按同一套契约实现，Flutter 层只写一份解析。

### Q55：Monorepo/Package 化组织有什么经验？

**回答：**
我会先区分 API 包、feature/implementation 包、example/test harness。API 包尽量稳定和零依赖，feature 包承载具体实现，example 用来验证接入路径。  
Monorepo 的价值是本地联调和统一测试方便，但也要避免包之间循环依赖。版本发布时要关注 tag、changelog 和消费方升级路径。

### Q56：Kotlin/Java、Jetpack、MVVM 你用在什么场景？

**回答：**
在德赛西威主要用于 Android 应用和平台互联服务，在 ADDX 用于 Android Native 能力、MethodChannel 实现、权限、BLE 状态、解绑参数等场景。  
MVVM 主要用于页面状态和业务逻辑解耦，Jetpack 相关能力用于生命周期、异步、状态和组件化。

### Q57：Android Framework、Binder IPC、Handler 经验怎么体现？

**回答：**
在车载项目里接触过 Framework 层服务和 TBoxService，涉及系统服务通信、线程调度、消息处理和跨进程调用。  
这类经验让我在做 IoT Native 能力时更关注生命周期、线程、回调、资源释放和系统权限，而不是只看 Flutter 页面。

### Q58：JNI/HAL 经验有多深？

**回答：**
我有项目使用和联调经验，理解 JNI/HAL 在 Android 系统和硬件能力之间的桥接作用。  
但我不会把自己包装成底层驱动专家。我的优势是能和底层能力对接，设计上层 SDK/Native API，并处理移动端接入、错误和生命周期。

### Q59：你熟悉哪些设计模式，实际怎么用？

**回答：**
代理模式常用于宿主 Adapter，把宿主能力代理给 SDK。  
单例用于配置、通道管理等全局状态，但要注意线程安全和测试替换。  
工厂/建造者用于创建复杂配置或 session。  
观察者用于事件流，比如绑定成功、状态变化。  
策略模式用于不同设备类型、渠道或固件能力差异。  
我更关注模式解决什么问题，而不是为了用模式而用模式。

### Q60：Ports & Adapters 和普通接口抽象有什么区别？

**回答：**
普通接口抽象可能只是为了方便 mock。Ports & Adapters 更强调架构边界：核心业务定义 Port，外部基础设施通过 Adapter 接入。  
核心业务不依赖具体网络库、数据库、埋点、Native 实现或宿主框架。这样 SDK 或服务在替换外部实现时，核心逻辑不变。

### Q61：Split API 适合什么场景？

**回答：**
适合多消费方、多 App、多模块复用，但不同消费方需要不同实现深度的场景。  
比如有的消费方只需要查询状态，有的需要完整 UI；有的 App 用 Snowplow，有的用别的埋点。Split API 可以让大家共享稳定契约，而不共享不必要依赖。

### Q62：你说熟悉 AI Agent 驱动研发，流程是什么？

**回答：**
一般是：先写需求和验收标准，再写设计和 ADR，明确边界；然后把任务拆小，让 AI Agent 按文件范围和验证命令执行；过程中用 quality gate、测试和 review 检查；最后做文档和代码漂移检查。  
核心是 AI 不能替代架构和验收，AI 只是在明确边界下执行。

### Q63：CLAUDE.md 架构约束写什么？

**回答：**
写项目里的硬规则，比如哪些包不能依赖哪些包、API 包必须零依赖、数据库 migration 规则、测试必须怎么跑、AI 不能跳过哪些验证、代码风格和目录边界。  
它相当于给 AI 和新人看的项目开发约束。

### Q64：quality_gate 是什么？

**回答：**
quality_gate 是把关键检查变成可执行门禁，比如测试是否跑过、文档是否同步、ADR 是否存在、golden data 是否匹配、接口契约是否漂移。  
它解决的是“review 靠人记忆容易漏”的问题。

### Q65：golden data 在 AI 开发里有什么作用？

**回答：**
golden data 是固定输入输出样本，用来验证 AI 修改后没有破坏协议、字段和边界。  
比如接口响应、Dart model 解析、订单号校验、配置解析，都可以用 golden data 固定预期。AI 很容易改字段名或漏边界，golden data 能快速暴露。

### Q66：Doc-Drift Gate 是什么？

**回答：**
它检查文档和代码是否漂移。比如 ADR 说 API 包零依赖，但代码引入了 UI 依赖；设计文档说有某个错误码，但代码没实现；或者代码新增能力但文档没有入口。  
AI 参与后，doc drift 很常见，所以需要专门检查。

### Q67：多轮分级 review 怎么做？

**回答：**
不是一轮看完所有东西。第一层看需求、设计、ADR 和文档规范；第二层看代码行为、安全、数据、性能、错误处理；第三层看测试、观测、发布、回滚闭环。  
不同级别的问题分 Critical、Important、Minor。Critical 必须修，Important 要修或明确接受风险。

## 8. 工程化与质量相关

### Q68：GitLab CI/CD 你主要做什么？

**回答：**
主要是把构建、测试、lint、质量门禁和部署检查串起来。  
我关注的是 CI 能不能覆盖真实风险，比如 SDK 的契约测试、后端接口测试、E2E、配置和观测验证，而不是只做 build。

### Q69：Jenkins 和 GitLab CI 都用过，区别是什么？

**回答：**
Jenkins 更灵活，适合历史项目和复杂自定义流水线，但维护成本高。GitLab CI 和代码仓结合更紧，MR、pipeline、review 更自然。  
实际选择看团队基础设施。我更关注流水线是否能承载质量门禁。

### Q70：SonarQube 主要看什么？

**回答：**
主要看静态质量、复杂度、重复代码、潜在 bug、安全风险和覆盖率趋势。  
但 SonarQube 不能代替业务 review，它更像底线工具。业务正确性、架构边界和跨端契约还需要人工和专项测试。

### Q71：Playwright E2E 用在什么地方？

**回答：**
主要用于 Web 管理后台或关键用户路径的端到端测试，比如 Warranty Plus 审核后台的操作流程、状态变化和表单校验。  
它能防止 AI 或重构后只保证接口正确，但页面流程断掉。

### Q72：GTest 用在哪？

**回答：**
GTest 更偏 C/C++ 或 native 层测试。在我过往经验里，它用于底层逻辑、协议或 native 组件的单元验证。  
在简历里我把它放在工程化与质量，是说明我接触过移动端以外的测试体系。

## 9. 项目取舍与不足

### Q73：Provisioning Kit 还有什么不足？

**回答：**
第一，native 底层演进快，虽然上层契约稳定，但历史方案切换带来一定学习成本。  
第二，强硬件场景自动化测试天然困难，Fake 能覆盖大部分逻辑，但最终仍需要真机 smoke。  
第三，部分宿主 Adapter 的质量取决于接入方，需要提供更完善的接入模板和检查清单。

### Q74：Homebase 项目有什么不足？

**回答：**
一是可观测性还可以更细，比如固件门控拦截人数、换网成功率、BLE 失败原因漏斗。  
二是跨端文案和多语言需要持续和运营/翻译平台同步。  
三是固件边界条件曾经出现过 off-by-one 这类问题，说明版本比较和边界测试必须前置。

### Q75：Warranty Plus 有什么不足？

**回答：**
一是 AI 辅助全栈交付对质量门禁依赖很高，如果门禁不完善，容易漏边界。  
二是审核后台 v1 可能更偏运营基本能力，后续可以增强批量处理、审计和分析。  
三是多 App 复用会带来宿主差异，需要持续维护接入文档和兼容策略。

### Q76：你做过失败或返工吗？

**回答：**
有。AI 辅助开发里比较典型的问题是 AI 容易只做 happy path，或者代码实现改了但文档没同步。  
我的改进是把这些失败模式沉淀成 quality gate 和 review 检查项。比如需求到测试映射、文档漂移检查、golden data、review 分层，都是从实际问题里演进出来的。

### Q77：你如何面对“AI 写代码不可靠”的质疑？

**回答：**
我同意 AI 直接裸写不可靠。所以我的做法不是相信 AI，而是限制 AI。  
先有设计和边界，再有任务拆分；先有测试和 golden data，再让 AI 修改；最后用 review 和 gate 验证。  
我不会因为是 AI 写的就降低标准，反而会提高可验证性要求。

## 10. 行为面试与管理协作

### Q78：你怎么推动别人配合 4 仓锁步？

**回答：**
先把问题从“大家帮我改代码”变成“如果不锁步会出现什么线上风险”。然后定义统一契约和合入顺序，让每个仓知道自己要改什么、什么时候合、怎么验收。  
我会用 ADR 或对齐文档记录决策，避免只靠口头同步。遇到阻塞时，优先保证兼容，比如先加 optional 参数、旧逻辑继续可用，再逐步切流。

### Q79：你和后端、测试、产品怎么协作？

**回答：**
和产品先对齐用户路径、非范围和验收标准；和后端对齐接口、错误码、数据模型和兼容；和测试对齐主路径、失败路径、边界和回归范围。  
我习惯用文档和表格把这些对齐下来，比如 AC、契约、状态表、测试映射，而不是只在群里说。

### Q80：你如何处理需求变化？

**回答：**
先判断变化影响的是需求、设计、接口、数据、测试还是发布。小变化可以局部调整；如果影响架构边界或长期维护，就补 ADR 或更新设计。  
我不倾向于直接改代码，因为跨端项目里直接改容易造成某端漏改。

### Q81：你如何判断一个功能是否应该 SDK 化？

**回答：**
看三个条件：是否多个 App 或模块复用；是否有稳定业务能力边界；是否能通过接口隔离宿主差异。  
如果只是一次性页面或强宿主定制，SDK 化可能过度。如果是设备绑定、保修延保这种长期复用能力，就适合 SDK 化。

### Q82：你如何避免过度设计？

**回答：**
我会要求每个抽象都有明确消费方和变化点。比如 Split API 是因为有三 App 和只读消费方；Ports 是因为宿主网络、导航、埋点确实不同；ADR 是因为决策会影响长期维护。  
如果没有明确变化点，就先用简单方案，并在 ADR 里写清楚为什么暂不扩展。

### Q83：你希望下一份工作做什么？

**回答：**
我希望继续做移动端或跨端架构相关工作，尤其是 Flutter、SDK、IoT、App + 设备 + 云端协同这类复杂链路。

## 14.1 Android 技术问答

### Q93：Android 蓝牙权限在不同版本上有什么差异？

**回答：**
Android 12 之前，蓝牙扫描经常和定位权限绑定，很多场景需要 `ACCESS_FINE_LOCATION`。Android 12 之后新增了 `BLUETOOTH_SCAN`、`BLUETOOTH_CONNECT`、`BLUETOOTH_ADVERTISE` 等运行时权限，语义更细。  
IoT 设备绑定里不能只判断“有没有蓝牙权限”，还要区分蓝牙开关是否打开、权限是否授权、系统版本需要哪个权限、用户是否选择了不再询问。  
我在项目里会把“权限状态”和“蓝牙适配器状态”拆开判断，避免把蓝牙未开误判成权限未授权。

### Q94：Android 蓝牙未开启和权限未授权，为什么容易混淆？

**回答：**
因为对业务层来说，两者都可能表现为扫描失败、连接失败或系统返回不可用，但用户引导完全不同。  
蓝牙未开启应该引导用户打开蓝牙开关；权限未授权应该引导用户授权或进入设置页。  
如果统一弹“请开启权限”，用户明明只是蓝牙没开，却被带去权限页，体验很差。我的做法是 Permission Guard 双轨判断：一条检查运行时权限，一条监听 `BluetoothAdapter` 状态。

### Q95：你怎么设计 Android 端 MethodChannel？

**回答：**
我会先从 Flutter 侧定义统一契约：method 名称、参数结构、返回结构、错误码、是否异步、取消行为。Android 端只按这个契约实现，不把 Android 内部类或异常直接暴露给 Flutter。  
错误统一用结构化错误，比如 code、message、details，避免 Flutter 层解析原生异常字符串。  
同时 Android/iOS 要共用同一份契约文档，否则最容易出现双端返回结构不一致。

### Q96：MethodChannel 调用中如何处理线程问题？

**回答：**
MethodChannel 回调到 Flutter 时要注意线程和生命周期。耗时操作不能跑在主线程，比如 BLE 扫描、网络、文件读取都要切后台线程或协程；但 UI 相关结果回调要回到主线程。  
另外页面销毁或绑定流程取消后，native 还继续回调会导致状态错乱，所以需要 session id 或 task id 绑定一次调用上下文，回调前检查当前 session 是否仍有效。

### Q97：Android Handler 你怎么理解？

**回答：**
Handler 是 Android 消息机制的一部分，配合 Looper 和 MessageQueue 做线程间消息调度。主线程有主 Looper，Handler 可以把任务投递到对应线程执行。  
在车载和 IoT 场景里，很多回调来自系统服务、蓝牙、native 或后台线程，需要切回主线程更新状态。  
风险是内存泄漏和延迟消息误触发，所以页面销毁时要移除 callback，避免 Handler 持有 Activity 或 ViewModel。

### Q98：Binder IPC 你怎么理解？

**回答：**
Binder 是 Android 主要的跨进程通信机制，Framework 层服务、系统服务、App 进程之间很多都是通过 Binder 通信。  
我在车载项目里接触过 Framework 层 TBoxService 这类服务，对 Binder 的理解主要体现在：接口要稳定、调用不能阻塞、异常要处理、跨进程对象生命周期要谨慎。  
这类经验让我在设计 Native SDK 或设备服务接口时，会更关注线程、回调、超时和资源释放。

### Q99：Android Service 和 SDK 能力边界怎么划分？

**回答：**
Service 更适合承载长期运行、跨页面、与系统或设备状态相关的能力，比如车载 TBoxService、设备连接状态、后台任务。  
SDK 更适合提供可复用业务能力，比如设备绑定、保修延保、通道封装。  
如果能力需要长期存活、被多个进程或模块调用，可能放 Service；如果主要是 App 内复用，优先做 SDK 或模块化包。

### Q100：Android 端如何做资源释放？

**回答：**
IoT 场景特别要注意 BLE scanner、连接对象、回调监听、Handler 消息、协程任务、BroadcastReceiver。  
绑定流程结束、页面销毁、用户取消、超时失败时，都要统一走 cleanup。  
我倾向于用 session 管理一次绑定生命周期，所有资源挂到 session 上，session 结束时集中释放，避免散在多个页面里遗漏。

### Q101：Android MVVM 在你的项目里怎么用？

**回答：**
MVVM 的核心是让 View 只负责展示和用户事件，ViewModel 管状态和业务编排，Repository 或 Service 负责数据和外部依赖。  
在设备绑定里，ViewModel 会维护当前步骤、权限状态、扫描状态、错误状态和按钮可用状态。  
但复杂 IoT 链路不能把所有逻辑塞进 ViewModel，底层绑定 session、Port、Adapter 还是要拆出去，否则 ViewModel 会变成大泥球。

### Q102：Android 端如何处理老版本兼容？

**回答：**
主要看三类兼容：系统版本、固件版本、SDK/API 版本。  
系统版本决定权限和系统 API；固件版本决定设备是否支持新协议；SDK/API 版本决定 Flutter 和 Native 是否能互相理解。  
我的做法是显式门控：系统能力做版本判断，固件能力做语义版本比较，MethodChannel 参数尽量 optional 向后兼容。

## 14.2 Flutter 技术问答

### Q103：Flutter SDK 和普通 Flutter 页面开发有什么区别？

**回答：**
普通页面开发更多关注页面展示和交互，SDK 开发要关注复用边界、宿主注入、依赖控制、版本兼容和接入成本。  
SDK 不能随便依赖宿主 App 的网络库、路由、主题、埋点和状态管理，否则换一个 App 就不能复用。  
所以我会把稳定 API、业务实现、UI、宿主 Adapter 分开设计，让宿主只实现必要 Port。

### Q104：Flutter Package 化时如何避免循环依赖？

**回答：**
先定义依赖方向：API 包在最底层，只放接口、值对象、错误类型；feature 包依赖 API 包，实现业务；宿主 App 依赖 API/feature 并提供 Adapter。  
API 包不能反向依赖 feature，feature 不能依赖宿主 App。  
如果发现需要从底层调用宿主能力，就定义 Port，由宿主注入，而不是让底层 import 宿主代码。

### Q105：为什么 Split API 对 Flutter SDK 很重要？

**回答：**
因为 Flutter SDK 往往有多种消费方：完整 App 要 UI，只读模块只要状态，测试工具只要 fake 能力。  
如果所有东西放一个包里，只读消费方也会被迫引入 UI、SharedPreferences、埋点、Native plugin 等依赖。  
Split API 把稳定契约拆出来，可以降低依赖污染，也让版本升级更安全。

### Q106：Flutter MethodChannel 常见问题有哪些？

**回答：**
第一，双端返回结构不一致。第二，错误码不统一。第三，异步回调丢失或重复。第四，页面销毁后仍回调。第五，参数变更没有兼容旧版本。  
解决方式是契约先行，所有方法定义 request/response/error，新增字段尽量 optional，破坏性变更必须升版本或做兼容分支。

### Q107：Flutter 如何设计错误码？

**回答：**
我会按来源分命名空间，比如 SDK 内部错误、Native 核心错误、后端错误。  
这样排障时能快速知道问题发生在哪层。错误码还要稳定，不要把底层异常字符串直接透出给业务。  
错误对象最好包含 code、message、source、recoverable、details，方便 UI 决定是重试、返回、提示用户还是上报。

### Q108：Flutter 状态管理你怎么看 GetX、Provider、Bloc、Riverpod？

**回答：**
我不会只按喜好选。GetX 上手快，但容易把状态、路由、依赖注入混在一起；Provider 简单稳定；Bloc 更强调事件和状态，适合复杂流程；Riverpod 在依赖组织和测试上更清晰。  
SDK 场景里我更谨慎引入强状态框架，因为会污染消费方依赖。能用原生 `ChangeNotifier`、`ValueNotifier`、`Stream` 或内部状态机解决时，不一定要引入重框架。

### Q109：Flutter 页面如何处理 loading、empty、error、success 状态？

**回答：**
我会把状态显式建模，而不是用多个 bool。比如 loading、empty、content、error、permissionDenied、offline。  
IoT 绑定里状态更多，还要区分权限未授权、蓝牙未开、设备未找到、固件不支持、网络失败、云端失败。  
状态模型清楚后，UI 才不会出现多个状态叠加或错误提示不准确。

### Q110：Flutter SDK 如何做测试？

**回答：**
分层做。纯 Dart 逻辑用 unit test；Port/Adapter 契约用 fake 或 mock；UI 用 widget test；关键路径用 integration/E2E。  
设备绑定强依赖硬件，不能所有测试都依赖真机，所以要有 FakeSessionPort、FakeClock、FakeBackend 这类测试装配。  
真机测试保留给少量 L4 smoke，验证 Native artifact 和真实设备链路。

### Q111：Flutter 和 Native 的版本如何兼容？

**回答：**
Flutter 调用 Native 时要考虑 App 包内 Native 版本和 Dart 包版本是否匹配。  
新增 MethodChannel 方法时，要有能力探测或 optional 参数；旧 Native 不支持时，Flutter 要降级或走旧流程。  
对于新硬件能力，还要叠加固件版本门控，不能只看 App 版本。

### Q112：Flutter SDK 如何处理宿主主题和多语言？

**回答：**
主题尽量消费宿主提供的 Theme 或通过 Port/配置注入，避免 SDK 写死颜色和字体。  
多语言可以由 SDK 自带 ARB，也可以由宿主注入文案。选择取决于多 App 是否共用文案、运营是否需要动态调整。  
我倾向于核心文案可由宿主覆盖，SDK 提供默认值，避免接入方必须一次性补齐所有文案。

### Q113：Flutter 中如何处理弱网和重试？

**回答：**
先区分用户动作和后台动作。用户提交类动作要避免重复提交，需要按钮状态、幂等 key 或后端去重；后台拉配置可以缓存和降级。  
重试不能无限重试，必须有上限、退避和错误分类。  
对用户来说，弱网时最重要是可恢复：保留输入、给明确提示、允许重试，而不是白屏。

## 14.3 iOS 技术问答

### Q114：你在 iOS 上主要做过哪些类型的工作？

**回答：**
主要是跨端 SDK 和 Native 能力协同，不是纯 iOS 页面专家。  
在 Homebase 项目里，我参与 iOS Core SDK 与 Flutter 绑定链路协同，包括 BLE 绑定能力、MethodChannel 双端契约、固件门控、解绑保留配对等。  
我的价值是保证 iOS 与 Android/Flutter 的能力和参数一致。

### Q115：iOS BLE 相关开发要注意什么？

**回答：**
要注意权限、蓝牙状态、扫描生命周期、连接超时、后台限制、回调线程和资源释放。  
iOS 的 CoreBluetooth 是异步回调模型，如果状态机设计不清楚，很容易出现重复连接、回调乱序或用户取消后仍继续绑定。  
所以同样适合用 session 管理一次绑定过程，把 scan、connect、discover、write、notify、timeout 都归属到同一上下文。

### Q116：iOS Local Network 权限和蓝牙权限有什么区别？

**回答：**
蓝牙权限主要影响 BLE 扫描和连接，Local Network 权限影响局域网发现和访问，比如某些设备发现、Bonjour/mDNS 或局域网通信场景。  
用户感知上都是“找不到设备”，但根因不同。  
设计引导时必须区分，否则用户不知道该开蓝牙、授权本地网络，还是检查 WiFi。

### Q117：iOS 与 Flutter 通信如何保证和 Android 一致？

**回答：**
不要让 iOS 和 Android 各自自由发挥。先定义 MethodChannel 契约，包括 method、参数、返回结构和错误码。  
iOS 端用 OC/Swift 实现同一套接口，Android 端也实现同一套接口。Flutter 层只依赖契约，不判断平台差异。  
确实存在平台差异时，要在契约里显式表达，比如 capability、unsupported reason，而不是让 Flutter 猜。

### Q118：OC 和 Swift 混编有什么注意点？

**回答：**
要注意桥接头、可见性、命名、可空性和类型映射。Swift 的 optional、enum、struct 到 OC 暴露时要谨慎设计。  
如果是 SDK 对外接口，接口要稳定，不能随便暴露内部 Swift 类型。  
我更倾向于对跨端通道暴露简单稳定的数据结构，比如 dictionary/json model，而不是暴露复杂对象。

### Q119：iOS SDK 设计如何避免宿主污染？

**回答：**
和 Flutter SDK 类似，要区分 SDK 稳定能力和宿主能力。SDK 不应该直接依赖宿主 App 的路由、埋点、账号系统或 UI 框架。  
需要宿主提供的能力通过 delegate、protocol 或 adapter 注入。  
这样 SDK 可以在多个 App 复用，也方便测试。

### Q120：iOS 端如何处理线程和回调？

**回答：**
BLE、网络和 SDK 内部处理可能在后台队列，UI 更新必须回主线程。  
另外异步回调要避免 retain cycle，block 里注意 weak self。  
绑定流程取消或页面释放后，要取消扫描、断开连接、清理 delegate，避免回调打到已经释放或无效的对象。

### Q121：iOS 端如何做版本和固件门控？

**回答：**
和 Android 类似，要区分 App 版本、SDK 版本、设备固件版本。  
新硬件能力不能只靠 App 发版打开，还要判断设备固件是否支持。  
版本比较要做语义版本处理，不能简单字符串比较，否则 `1.10.0` 和 `1.2.0` 会出错。

### Q122：你如何回答“iOS 不是你的强项”？

**回答：**
我会承认我不是纯 iOS 专家，但我有 iOS Native SDK 协同和跨端契约经验。  
我的强项不是单独写复杂 iOS UI，而是把 iOS 放在 Android/Flutter/Native SDK 的整体链路里，保证能力一致、契约一致、版本一致。  
如果岗位要求纯 iOS 深度，我需要补；如果要求跨端 SDK 和 IoT 链路协同，我是匹配的。

## 14.4 后端技术问答

### Q123：你后端能力的真实边界是什么？

**回答：**
我能做后端接口设计、数据模型设计、分层架构、联调、测试、部署和可观测性设计，也能借助 AI Agent 完成 Go/Next.js 的实现和排查。  
但我不是传统意义上多年手写后端的专家，复杂分布式治理、高并发中间件内核调优不是我的主场。  
我的优势是把后端作为移动端业务闭环的一部分，设计清楚接口、信任边界、数据流和验收标准。

### Q124：Warranty Plus 后端为什么适合 Ports & Adapters？

**回答：**
因为业务核心是保修申请、审核、订单号校验、配置和渠道策略，这些逻辑不应该直接依赖 MySQL、HTTP 框架或第三方平台。  
Ports & Adapters 可以让核心 service 依赖抽象，比如 WarrantyRepo、OrderValidator、EventTracker、FeatureFlag。MySQL、Snowplow、GrowthBook 都是 adapter。  
这样逻辑层更容易测试，外部依赖替换也更安全。

### Q125：你如何设计一个后端接口？

**回答：**
我会先从用户场景和状态变化出发，而不是先定 URL。  
然后定义 request、response、错误码、幂等策略、鉴权来源、权限边界、数据写入、可观测性和兼容策略。  
最后补测试：成功、失败、边界、重复提交、旧客户端兼容。

### Q126：后端鉴权和信任边界怎么设计？

**回答：**
先明确身份从哪里来，比如 Gateway JWT、session、HMAC admin token，而不是信任客户端 body 里的 userId。  
业务服务只消费可信上下文，不重复造身份系统。  
对于 Warranty 这类服务，可以让 Gateway 或宿主预过滤承担身份和设备归属的一部分，服务内部负责业务登记和审核，但这个边界必须文档化和测试覆盖。

### Q127：MySQL 双表模型如何设计？

**回答：**
先区分事实和流程。用户提交的保修申请是事实记录，运营审核是处理流程。  
申请表记录 user、device、order、channel、submitted_at 等；审核表记录 review_status、reviewer、review_reason、reviewed_at 等。  
这样可以避免一张表混合用户事实和运营动作，也方便审计和后续扩展。

### Q128：MySQL schema 设计你会关注什么？

**回答：**
主键、唯一约束、索引、状态字段、时间字段、软删、审计字段和兼容迁移。  
比如订单号或设备序列号要考虑唯一性和重复提交；列表查询要有索引；状态字段要能表达审核流程；时间字段要区分提交、审核、过期。  
另外 migration 要考虑旧版本代码是否还能读写，不能只看新代码。

### Q129：如何处理重复提交？

**回答：**
前端要禁用重复点击，但不能只靠前端。后端要有幂等设计，比如唯一键、幂等 key、状态 guard。  
如果用户重复提交同一个订单号和设备，后端应该返回已有状态或明确冲突，而不是生成多条重复记录。  
并发场景要依赖数据库唯一约束兜底。

### Q130：如何设计订单号校验？

**回答：**
先做本地格式校验，比如不同渠道订单号格式、长度、字符集。  
再做服务端业务校验，比如是否已提交、是否已审核、是否属于支持渠道。  
如果未来接第三方订单系统，还要考虑接口超时、重试、降级和对账。v1 如果只是格式和业务库校验，要明确边界，不夸大成真实电商订单校验。

### Q131：后端错误码怎么设计？

**回答：**
错误码要面向客户端可处理。常见分类包括 validation、auth、permission、conflict、not_found、dependency、internal。  
每类错误要对应用户体验，比如参数错让用户修改，权限错拒绝或引导登录，冲突提示已提交，依赖失败提示稍后重试。  
不要把数据库错误或 panic 直接返回给客户端。

### Q132：后端如何做可观测性？

**回答：**
至少要有结构化日志、指标和必要 trace。  
日志记录 action、result、reason、correlation_id；指标记录请求量、错误率、延迟、业务成功/失败；dashboard 能回答当前是否异常、影响多少用户、失败在哪一步。  
对 Warranty Plus，重点关注提交量、审核量、订单号校验失败、配置降级、接口错误率和延迟。

### Q133：监控指标设计要注意什么？

**回答：**
标签基数要受控，不能把 userId、orderId、email 这种高基数或敏感字段放 label。  
counter 用于次数，histogram 用于延迟，gauge 用于当前状态或队列深度。  
业务失败原因要枚举化，方便聚合和告警。

### Q134：后端配置热更新有什么风险？

**回答：**
配置错可能比代码 bug 影响更快，比如入口关闭、渠道缺失、实验异常。  
所以配置要有默认值、schema 校验、缓存、回滚、熔断和监控。  
客户端也要能处理配置获取失败，不能因为配置平台异常就白屏。

### Q135：后端部署和回滚你会关注什么？

**回答：**
关注发布顺序、数据库 migration、配置、secret、环境差异、健康检查、灰度和回滚后数据兼容。  
如果新版本写了旧版本不认识的数据，回滚会有风险。  
所以设计时要考虑 old app + new schema、new app + old schema、rollback 后新数据怎么处理。

### Q136：你如何用 AI 辅助后端开发但保证质量？

**回答：**
先写接口契约、数据模型、状态和测试要求，再让 AI 实现。  
实现后必须跑测试、做 contract/golden data 校验、review 安全和数据一致性、检查可观测性。  
AI 适合加速样板代码和测试补齐，但架构边界、信任边界和验收标准必须由人定义。

### Q137：如果面试官现场问 Go 细节，你怎么回答？

**回答：**
会如实回答我熟悉项目级 Go 服务设计和 AI 辅助实现，但不是长期深耕 Go runtime 的专家。  
常规的 handler、service、repo、interface、context、error handling、testing 我能讲；如果问 GMP、内存模型、极限并发调优，我会说这不是我的主要经验。  
这样比硬答更可信。

## 14.5 跨端技术综合问答

### Q138：Android、iOS、Flutter、后端一起改时，你怎么控制一致性？

**回答：**
先定契约，再分端实现。契约包括 API、MethodChannel、错误码、状态机、固件门控、数据字段和测试用例。  
然后每端按同一份契约实现，最后用 E2E 或联调 checklist 验证。  
如果不能同时上线，就要设计兼容，比如 optional 字段、feature flag、版本门控和降级路径。

### Q139：跨端功能怎么设计测试矩阵？

**回答：**
按“平台 × 场景 × 状态”建矩阵。平台包括 Android/iOS/Flutter/后端；场景包括主路径、权限拒绝、蓝牙未开、网络失败、固件不支持、重复提交；状态包括新旧版本、新旧固件、在线离线。  
不是每个组合都做自动化，但高风险组合必须覆盖，低风险组合可以人工 smoke。

### Q140：你如何判断问题出在 App、Native、设备还是后端？

**回答：**
靠分层信号。Flutter 层看页面状态和 MethodChannel 请求；Native 层看 BLE/WiFi/session 日志；设备侧看固件版本和协议响应；后端看 request id、接口日志和业务指标。  
所以设计时就要有 correlation id、错误来源和结构化错误码。  
没有这些信号，排障只能靠猜。

### Q141：IoT 设备绑定链路最难的是什么？

**回答：**
最难的是不确定性：用户权限、蓝牙状态、WiFi 环境、固件版本、设备状态、云端接口、App 生命周期都可能出问题。  
所以绑定链路不能只写 happy path，必须有状态机、超时、取消、重试、降级、错误分类和观测。  
这也是我为什么强调 SDK 架构和 session API。

### Q142：如何设计一个设备绑定状态机？

**回答：**
先列状态：idle、checkingPermission、scanning、connecting、fetchingWifi、binding、success、failed、cancelled、timeout。  
再定义每个状态允许的事件和转换，比如 cancel 在哪些状态有效，timeout 后是否能 retry，success 后是否允许重复回调。  
最后给每个关键转换加日志和指标，排障时能知道卡在哪一步。

### Q143：你如何解释“移动端架构”不是空话？

**回答：**
移动端架构不是画图，而是解决依赖、边界、版本、测试和发布问题。  
比如 API 包不能污染依赖，Native 通道要统一契约，宿主能力要通过 Port 注入，SDK 要能在多个 App 复用，硬件能力要有固件门控，AI 产物要有质量门禁。  
这些都是真实影响交付和维护成本的架构问题。

### Q147：Homebase 的“量产目标”为什么可以写进简历？

**回答：**
因为这个项目不是单纯页面开发，它直接服务于新硬件 HB1 的发布、上架和售卖。相机绑 Hub、换网、解绑保留配对、固件校验都是售卖前必须闭环的核心能力。  
如果这些能力没有 App 侧支持，新硬件即使设备和固件准备好了，也无法给用户完整使用路径。  
所以我会把收益讲成“支撑新型号按期完成 App 侧闭环、发布上架与量产售卖”，而不是只说“完成几个页面”。

### Q148：Homebase 的“相机绑 Hub”是什么？

**回答：**
传统绑定可能是相机直接绑定到用户账号或 App。Homebase 模型里，Hub 是中枢，相机作为子设备绑定到 Hub，形成 1 个 Hub 管多台相机的拓扑。  
App 侧要处理的不只是新增一个绑定入口，还包括选择 Hub、校验 Hub 和相机能力、固件版本门控、绑定结果展示、设备卡片展示，以及后续解绑或换网。  
这也是为什么它涉及 Flutter、Android、iOS 和 iOS Core SDK 多仓协同。

### Q149：Homebase 里的“设备卡片”有什么技术点？

**回答：**
设备卡片表面是 UI，但背后体现设备模型变化。相机从独立设备变成 Hub 下的子设备后，卡片要表达父子关系、在线状态、固件能力、入口能力和异常状态。  
如果模型没有设计清楚，UI 会出现新旧设备混展示、能力入口误展示、固件不支持却展示新功能等问题。  
所以设备卡片的重点是“基于设备型号、固件版本和能力开关控制功能暴露”。

### Q150：为什么“固件版本门控”能降低发布风险？

**回答：**
App 发版和固件升级不是完全同步的。用户可能拿着新 App 连接旧固件，也可能旧 App 遇到新设备。  
固件版本门控就是在展示和执行功能前判断设备是否支持某项能力，不支持就隐藏、降级或提示升级，避免把旧固件不认识的命令下发给设备。  
这能降低“新功能上线后，旧设备被误导进入不可用路径”的风险。

### Q151：Permission Guard、SingleTask、固件版本门控三者分别解决什么？

**回答：**
Permission Guard 解决系统能力前置条件，比如蓝牙未开启、权限未授权、系统能力不可用，避免用户走进不可恢复路径。  
SingleTask 解决绑定流程的任务隔离，避免重复点击、页面回退、异步回调导致流程错乱。  
固件版本门控解决设备能力差异，确保新旧硬件、新旧固件并存时，App 不会暴露设备不支持的功能。  
这三者分别对应“系统条件、流程状态、设备能力”三类风险。

### Q152：Warranty Plus 为什么强调“1 个月完成全链路交付”？

**回答：**
因为这个项目不是单端功能，而是 App 入口、Flutter SDK、Go 服务、MySQL 数据模型、Next.js 审核后台、埋点、Grafana 看板、部署联调一起交付。  
1 个月完成说明我不是只做移动端页面，而是能在 AI Agent 协助下，把方案设计、实现、联调、验收和上线闭环组织起来。  
回答时要注意边界：实现上有 AI 辅助，但方案约束、接口验收、核心审查、质量门禁和发布风险控制由我负责。

### Q153：Warranty Plus 的“上线后 0 事故”怎么讲才不空？

**回答：**
0 事故不是说没有任何小问题，而是没有造成线上主流程中断、错误审核、数据不可用或大面积用户影响。  
这个结果来自几个控制点：接口和数据模型提前对齐，App 有弱网缓存和配置兜底，运营后台能处理审核状态，埋点和 Grafana 看板能观察使用状况和异常趋势。  
如果面试官追问证据，我会结合“上线后通过看板发现小问题，并在客诉前快速处理”来讲，不把它说成绝对没有 bug。

### Q154：Warranty Plus “通过看板在客诉前解决小问题”具体是什么意思？

**回答：**
意思是我们不是等用户投诉才知道问题，而是通过埋点和 Grafana 看板观察关键链路，比如入口曝光、申请提交、订单号填写、设备选择、审核状态、错误原因、接口错误率。  
当某个错误码、某个 App、某个版本或某个阶段异常升高时，可以提前定位是配置、接口、SDK 还是运营后台问题。  
这类看板的价值是把“线上不可复现问题”变成“有数据、有范围、有阶段”的排查问题。

### Q155：Warranty Plus 的“SDK 健康度、使用状况、问题监测”分别看什么？

**回答：**
SDK 健康度看 SDK 是否正常工作，比如入口是否可用、配置是否拉取成功、核心接口错误率、页面是否能进入、提交流程是否完成。  
使用状况看用户和运营是否真的在用，比如入口曝光、点击、提交量、审核量、不同 App/版本/渠道的分布。  
问题监测看失败原因，比如订单号格式错误、设备不匹配、网络失败、配置降级、接口超时、审核状态回写异常。  
三者合起来，才能判断“功能上线了”是不是等于“功能健康地被使用”。

### Q156：Snowplow 和 Grafana 在简历里分别代表什么？

**回答：**
Snowplow 更偏用户行为和业务事件埋点，比如入口曝光、按钮点击、提交流程、失败原因、功能使用分布。  
Grafana 更偏看板和监控展示，可以接业务指标、服务指标或日志统计，帮助研发和运营观察趋势、定位异常。  
可以简单理解：Snowplow 负责把业务事件采集上来，Grafana 负责把关键指标可视化出来；具体数据源可以根据公司内部平台而定。

### Q157：GrowthBook 为什么会出现在 Warranty Plus 技术栈里？

**回答：**
GrowthBook 通常用于 feature flag 或实验控制。Warranty 这类能力可能需要按 App、版本、渠道、地区或用户分组控制入口开关。  
它的价值是降低发版风险：功能可以灰度开启，也可以在异常时快速关闭或切换策略。  
但我会强调，GrowthBook 不是主业务逻辑本身，App 侧必须有默认值和兜底，不能因为配置平台异常导致主流程不可用。

### Q158：ArgoCD 在这个项目里你应该怎么讲？

**回答：**
我不会把自己包装成专职 DevOps。ArgoCD 在这里表示我理解 GitOps 部署链路：服务或配置通过 Git 作为声明式来源，由 ArgoCD 同步到环境。  
我的关注点是部署闭环：代码合入后怎么构建、怎么部署、怎么验证健康、怎么回滚，以及配置和数据库变更是否兼容。  
对移动端负责人来说，理解这些能帮助 App、后端和后台按同一节奏上线。

### Q159：Provisioning Kit 里的 melos 是什么？

**回答：**
melos 是 Flutter/Dart monorepo 管理工具，用来管理多个 package 的依赖、脚本、测试和版本。  
Provisioning Kit 这类 SDK 往往不是单包：可能有 API 包、feature 实现包、example、测试装配等。melos 可以让本地开发、统一测试和包间联调更顺畅。  
面试时不需要讲成“精通 melos 内部机制”，重点讲它解决了 Flutter 多包工程组织问题。

### Q160：AAR 是什么，为什么 Provisioning Kit 会用 AAR？

**回答：**
AAR 是 Android Library 的打包格式，里面可以包含 Kotlin/Java 代码、资源、Manifest 和 native 相关内容。  
Provisioning Kit 用 AAR，是为了把底层 Android native 配网能力作为稳定制品交付给 Flutter plugin 或宿主，而不是每个 App 各自复制底层实现。  
这让底层能力可以独立版本化，也方便 Native 团队或 SDK 团队维护统一实现。

### Q161：BLE、WiFi、Matter 在设备绑定里分别承担什么？

**回答：**
BLE 常用于近场发现、建立临时连接、传输 WiFi 信息或设备初始化。  
WiFi 是设备最终联网的主要通道，用户选择网络、输入密码、设备连接路由器都属于这个链路。  
Matter 是智能家居设备互联协议，涉及更标准化的设备发现、配网和生态兼容。  
在绑定 SDK 里，难点是这些协议不是孤立的，用户权限、设备固件、App 状态、云端注册都要一起编排。

### Q162：iOS Core SDK 在 Homebase 里是什么角色？

**回答：**
iOS Core SDK 承担 iOS 侧底层设备能力或统一接口的角色，Flutter 上层不应该直接了解所有 iOS 底层细节。  
Homebase 新能力如果只改 Flutter 页面，不改 iOS Core SDK 或 Native 能力，iOS 端可能无法真正执行相机绑 Hub、换网、解绑保留配对等动作。  
所以我在项目里强调四仓锁步：Flutter 负责页面和业务编排，Android/iOS 负责 Native 能力，iOS Core SDK 提供更底层或统一的设备接口。

如果问复杂 iOS UI 或 Swift 高级语言特性，我会承认不是主场。

### Q164：Next.js Admin 在 Warranty Plus 中做什么？

**回答：**
Next.js Admin 是运营审核后台，给运营查看保修申请、订单号、设备信息、用户账号和审核状态。  
它和 App、后端一起构成闭环：用户在 App 提交，Go 服务落库，运营在 Admin 审核，审核状态再回到 App 或后台数据里。  
所以它不是一个孤立后台页面，而是售后保修业务闭环的一部分。

### Q165：MySQL 数据模型在 Warranty Plus 里怎么讲？

**回答：**
可以从实体和状态讲：用户、设备、订单号、保修申请、审核状态。  
核心是区分用户提交的事实记录和运营审核的处理记录，避免把所有状态塞进一个混乱模型。  
还要考虑唯一约束、重复提交、状态回查、审核状态变更、时间字段和后续审计。

### Q166：为什么简历里写“自己负责方案约束、接口验收、核心审查”？

**回答：**
因为这个项目使用了 AI Agent 辅助实现后端和 Web，必须讲清楚人的价值边界。  
AI 可以帮助写代码、补测试、生成后台页面，但方案边界、接口契约、数据模型、异常处理、权限边界、联调验收和发布风险不能交给 AI 自己决定。  
这句话是在强调我不是“让 AI 乱写”，而是把 AI 放进受约束的工程流程。

### Q167：为什么“多 App 复用同一套 SDK、服务和后台”能节省约 2/3 成本？

**回答：**
如果三个 App 各自做保修能力，就需要三套入口、三套状态、三套接口适配、三套审核后台或运营对接，后续 bug 也要修三遍。  
共用一套 SDK、服务和后台后，差异主要集中在宿主配置、主题、入口开关和少量适配，核心流程和服务只维护一套。  
所以节省约 2/3 是按三个 App 避免重复建设估算的，不是精确财务数字。面试中要说明这是工程量估算。

### Q168：Provisioning Kit 的“SDK UI / 自定义 UI”两种模式怎么讲？

**回答：**
SDK UI 模式是接入方直接使用 SDK 提供的页面和流程，接入最快，适合 UI 差异不大的 App。  
自定义 UI 模式是接入方自己画页面，但复用 SDK 的流程编排、状态机、Native 通道和业务适配层，适合品牌或交互差异大的 App。  
这就是为什么简历里可以写：沿用 SDK UI 时接入工作量降低 70%-80%，自定义 UI 时核心能力复用、工作量减少 50%-60%。

### Q169：Provisioning Kit 的“追踪成功率、失败原因、耗时分布”为什么重要？

**回答：**
设备绑定失败原因很多：权限、蓝牙、WiFi、Matter、固件、云端、用户取消、网络波动。只看最终失败率无法定位问题。  
按 App、型号、固件、阶段、错误码追踪成功率、失败原因、耗时分布，就能看出问题集中在哪类设备、哪个固件、哪个流程阶段。  
这对 SDK 健康和后续优化非常关键，因为配网问题经常线上难复现，只能靠分层数据定位。
